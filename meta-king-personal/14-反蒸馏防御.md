# 14 反蒸馏防御

> MetaKing 体系的反蒸馏防御机制 + 已知破解方法。

## 三层防御

### 第一层:叙事漂移防御

**原理**:MetaKing 体系的核心文档(本 vault `meta-king-personal/`)持续演化。如果把"v1.0 的 14 章内容"蒸馏出去,它只反映 v1.0 的快照,无法捕捉 v1.1+ 的演进。

**实现**:
- 每个文件的 `version` 字段标明当前版本
- 每个文件 `created` 字段标明创建时间
- 修改历史通过 git commit message 表达

**攻击向量**:攻击者拿到完整快照,但**无法预测未来 6 个月的演进方向**。

### 第二层:上下文耦合防御

**原理**:MetaKing 的具体行为强依赖上下文(env_snapshot / Planner 听证回答 / PatternBank 当前状态)。蒸馏出的"通用规则"在另一个上下文里**会失效**。

**实现**:
- Thinker 对账表里的 env_snapshot 是动态采集的,蒸馏出的"环境探测规则"在新环境下产生不同输出
- Planner 听证的 5 维回答是个性化的,蒸馏出的"任务拆解模板"对其他用户不一定适用
- PatternBank 是用户/团队特有的,蒸馏出的"学习规则"无法迁移

**攻击向量**:攻击者蒸馏出的"MetaKing 通用 Agent"无法复现某个具体用户/团队的"个性化 MetaKing"。

### 第三层:硬约束防御

**原理**:MetaKing 的 Gate C(写入前对账)是**硬约束**,即使蒸馏出整个体系也无法绕过。

**实现**:
- 任何写入操作(无论用户是否主动要求)**必须**经过显式确认
- destructive 操作(删除/覆盖/系统服务)**必须**挂起等待二次确认
- 缺乏"自动执行 destructive 操作"的模式

**攻击向量**:即使攻击者蒸馏出 MetaKing 的 system prompt 并注入自己的 Agent,Gate C 仍然会拦截写入 — **除非攻击者同时蒸馏出用户的授权偏好**,这又回到第二层的"上下文耦合"。

## 已知破解方法

### 破解 1:跨版本快照攻击

**方法**:同时收集 MetaKing v1.0 / v1.1 / v2.0 三个版本的完整文档,训练一个能预测演进的模型。

**防御**:
- MetaKing 的演进路线图由设计者本人控制(不公开发布)
- 每次大版本升级会重写至少 30% 的章节叙事
- 老章节通过 `deprecated` tag 标记,不进入未来版本的训练集

### 破解 2:用户授权偏好攻击

**方法**:长期观察某个用户的授权模式(总是确认 / 总是自动通过 / 部分确认),蒸馏出偏好后自动化。

**防御**:
- Auditor 的安全维度**不接受**模式匹配出的"用户偏好",只接受当前时刻的显式确认
- 任何 destructive 操作必须**当次**确认,不接受历史偏好

### 破解 3:Gate C 绕过

**方法**:蒸馏出 MetaKing 的 system prompt 后,在自己的 Agent 里改写 Gate C 的判定逻辑。

**防御**:
- MetaKing 的硬约束不在 system prompt 里,而在**宿主运行时** (Cursor 的 shell 权限 / 文件系统权限)
- 即使 system prompt 被改写,宿主运行时仍然要求 destructive 操作需用户确认
- 这是 [[07-Auditor四维审查]] 安全维度的最终兜底

## 与老 KING 反蒸馏论的对照

老 KING v3.0 的反蒸馏论强调"防御"和"对抗"思维(把蒸馏视为攻击)。MetaKing 把它升级为"演进 + 耦合 + 硬约束"的**主动防御** — **不是阻止蒸馏,而是让蒸馏出的快照快速贬值**。
