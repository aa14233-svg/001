# 10 因果链追溯

> 每次决策写入因果链,保证可追溯。

## 形式化定义

```
C = { (input, action, output, timestamp) | action ∈ Actions }
```

每条因果链条目是一个四元组:

```json
{
  "task_id": "impl_redis_pool_001",
  "step": "Thinker.probe",
  "input": {
    "dag": [{"id": "impl_redis_pool", "depends": [], "agent": "Builder", "mode": "create"}],
    "env_snapshot": {...}
  },
  "action": "produce_gap_analysis",
  "output": {
    "satisfied": [...],
    "mismatch": [{"need": "redis-py", "have": "未安装", "fix": "pip install redis"}]
  },
  "timestamp": "2026-07-16T03:36:48Z",
  "duration_ms": 234
}
```

## 存储

```sql
-- PostgreSQL schema
CREATE TABLE causal_chains (
  id SERIAL PRIMARY KEY,
  task_id VARCHAR(64) NOT NULL,
  step VARCHAR(64) NOT NULL,
  input JSONB,
  action VARCHAR(128),
  output JSONB,
  timestamp TIMESTAMPTZ DEFAULT NOW(),
  duration_ms INT,
  INDEX idx_task_id (task_id),
  INDEX idx_timestamp (timestamp)
);
```

## 用途

### 1. 调试与回溯

```sql
-- 查 task_id 全部步骤
SELECT step, action, timestamp, duration_ms
FROM causal_chains
WHERE task_id = 'impl_redis_pool_001'
ORDER BY timestamp;

-- 查某类失败的根因
SELECT step, action, output
FROM causal_chains
WHERE output->>'status' = 'fail'
  AND action LIKE 'Builder.%'
  AND timestamp > NOW() - INTERVAL '7 days'
ORDER BY timestamp DESC;
```

### 2. 审计

- 谁在什么时间触发了什么操作
- 每个决策的依据(input)
- 每个决策的结果(output)

### 3. 回滚

当 [[07-Auditor四维审查]] 的恢复维度触发时:

```python
mk.rollback(task_id, strategy='auto')
# 1. 读取 causal_chains 中该 task 的所有写操作
# 2. 反向执行:删除文件 / git revert / 撤销配置
# 3. 在 causal_chains 写入 "rollback" 条目
```

## 与老 KING 因果逻辑链的对照

老 KING v3.0 的因果逻辑链定义完全相同(都是 input/action/output/timestamp 四元组),但 MetaKing 做了两项升级:

1. **绑定 task_id**: 老 KING 的因果链是单步记录,MetaKing 要求每条记录带 `task_id`,这样可以一次性回溯整个任务的完整链路。

2. **绑定 duration_ms**: 便于事后分析哪一步最耗时,优化 Planner 的 DAG 拆分粒度。
