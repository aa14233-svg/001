# Hook+Loop 工程约束与验收体系

> King Agent 不是"放手不管"，而是"全程监督"——通过 Hook 和循环约束确保每一步都符合标准。

## 一、Hook 工程体系

### 4 个核心 Hook 节点

```
任务开始 → Hook 1（准备）→ 执行 → Hook 2（执行中）
              │                              │
              ▼                              ▼
          安全检查                    进度/质量/风险
              │                              │
              ▼                              ▼
          资源确认                    阶段性检查
              │                              │
              ▼                              ▼
          依赖检查                    纠正/调整
              │                              │
              ▼                              ▼
       ┌─────┴─────┐               ┌─────────┴─────────┐
       │           │               │                   │
       ▼           ▼               ▼                   ▼
   Hook 3（验收）   完成        Hook 4（完成）         交付
       │                           │
       ▼                           ▼
   D1/D2/D3 验收              完整检查
       │                           │
       ▼                           ▼
   通过/驳回                  文档/结果
```

### Hook 1：任务准备

**时机**：任务分发前执行

| 检查项 | 内容 | 执行者 | 通过标准 |
|--------|------|--------|---------|
| 安全检查 | 路径是否在禁区，是否有高危操作 | SecurityAgent | 无高危操作 |
| 资源确认 | 文件是否存在，依赖是否满足 | SearchAgent | 资源可用 |
| 权限验证 | 是否有权限执行该操作 | SecurityAgent | 权限通过 |
| 上下文确认 | 是否理解任务上下文 | ChatAgent | 上下文清晰 |
| Provider 检查 | 当前 Provider 是否可用（联动 cc-switch） | SkillHubAgent | Provider 可用 |
| 预算检查 | Token 预算是否充足（联动 token-stats） | AuditAgent | 预算未超限 |

**输出**：通过/拒绝（附原因）

### Hook 2：执行中

**时机**：每阶段完成后执行

| 检查项 | 内容 | 执行者 | 通过标准 |
|--------|------|--------|---------|
| 进度检查 | 是否按计划推进 | PlanAgent | 进度偏差 < 20% |
| 质量检查 | 产出是否符合标准 | CodeReviewAgent | 质量标准通过 |
| 风险检查 | 是否有偏离风险 | TaskAgent | 无高风险项 |
| 资源检查 | 预算是否充足 | AuditAgent | Token 未超限 |
| Provider 检查 | Provider 是否正常工作 | SkillHubAgent | 响应正常 |

**输出**：通过/需要调整/需要回退

**检查示例**：
```
任务："创建 React 登录页面"
Stage 1：完成 LoginPage 组件

Hook 2 检查：
├── 进度：Stage 1 完成，符合计划
├── 质量：组件代码规范，功能正确
├── 风险：无
├── 资源：Token 消耗 150，预算充足
└── Provider：响应正常
→ 通过，进入 Stage 2
```

### Hook 3：验收

**时机**：子任务完成或全任务完成时执行

| 检查项 | 内容 | 执行者 | 通过标准 |
|--------|------|--------|---------|
| D1 验收 | 阶段子任务完成度 | PlanAgent | 子任务 100% 完成 |
| D2 验收 | 模块功能完整性 | CodeReviewAgent | 接口正常，功能完整 |
| D3 验收 | 全任务总体验收 | AuditAgent | 全部符合原始需求 |

**输出**：通过/驳回（附失败原因）

### Hook 4：完成

**时机**：全任务交付前执行

| 检查项 | 内容 | 执行者 | 通过标准 |
|--------|------|--------|---------|
| 完整检查 | 所有子任务是否完成 | TaskAgent | 无遗漏 |
| 文档检查 | 是否生成了必要的文档 | DocAgent | 文档完整 |
| 清理检查 | 临时文件是否已清理 | DeployAgent | 无残留 |
| Token 审计 | Token 消耗是否在预算内（联动 token-stats） | AuditAgent | 预算合规 |
| 引力盆更新 | 高频路径是否已固化 | PlanAgent | 已更新 |

**输出**：通过/交付/需要补充

## 二、三级验收体系（D1/D2/D3）

### D1：阶段验收

```
D1 验收标准
  │
  ├── 当前阶段任务全部完成
  │    └── 子任务完成率 100%
  │
  ├── 产出物经 Agent 自查
  │    └── 无语法错误、无逻辑缺陷
  │
  ├── 无重大错误或偏离
  │    └── 未偏离原始需求
  │
  └── 通过 → 进入下一阶段
      拒绝 → 返工修正（最多 3 次）
```

**调用时机**：每个子任务完成后

**示例**：
```
登录页面开发项目：

D1-1：LoginPage 组件完成
  ├── 通过：组件代码完整，功能正确
  └── 进入下一个 D1

D1-2：表单验证完成
  ├── 通过：验证逻辑完整
  └── 进入下一个 D1

D1-3：API 对接完成
  ├── 拒绝：API 请求格式有误
  ├── 修正 → 重验 → 通过
  └── 进入 D2
```

### D2：模块验收

```
D2 验收标准
  │
  ├── 相关模块的所有子任务完成
  │    └── 所有 D1 已通过
  │
  ├── 模块间接口正常
  │    └── 组件间通信、API 调用正常
  │
  ├── 模块功能符合预期
  │    └── 功能完整，符合需求
  │
  ├── 模块内无严重缺陷
  │    └── 无阻塞性 Bug
  │
  └── 通过 → 进入下一模块
      拒绝 → 缺陷修复（最多 2 次）
```

**调用时机**：相关功能模块完成后

**示例**：
```
登录页面开发项目：

D2：登录功能模块验收
  ├── 检查 LoginPage 组件
  ├── 检查表单验证
  ├── 检查 API 对接
  ├── 检查路由配置
  ├── 检查错误处理
  ├── 通过：所有功能正常
  └── 进入 D3
```

### D3：总体验收

```
D3 验收标准
  │
  ├── 全任务所有子任务完成
  │    └── 所有 D1/D2 已通过
  │
  ├── 所有模块集成正常
  │    └── 全功能联调通过
  │
  ├── 最终产出符合原始需求
  │    └── 逐项对照需求文档
  │
  ├── 所有 Hook 检查均通过
  │    └── Hook 1/2/3/4 全部通过
  │
  ├── Token 消耗在预算内
  │    └── token-stats 审计通过
  │
  └── 通过 → 交付
      拒绝 → 回归修复（人工介入）
```

**调用时机**：全任务完成后

**示例**：
```
登录页面开发项目：

D3 总体验收：
├── [通过] 所有子任务完成（4/4）
├── [通过] 模块集成正常
├── [通过] 符合原始需求
├── [通过] Hook 1/2/3/4 全部通过
├── [通过] Token 消耗 450/500 预算内
└── 通过 → 交付用户
```

## 三、Loop 循环机制

```
当验收不通过时，触发 Loop 循环：

D1 拒绝 → 修正 → 重新 D1 验收
  │
  ├── 最多 3 次循环
  ├── 每次循环记录失败原因
  │    └── 如：API 格式错误 → 记录到引力盆失败模式
  └── 3 次后升级到 D2 审查
       └── D2 判定是否继续或终止

D2 拒绝 → 修复 → 重新 D2 验收
  │
  ├── 最多 2 次循环
  ├── 每次循环记录失败原因
  └── 2 次后升级到 D3 审查
       └── D3 判定是否需人工介入

D3 拒绝 → 重大问题 → 人工介入
  │
  ├── 记录失败模式到引力盆
  ├── 生成失败报告
  └── 等待人工决策
```

### Loop 循环的 Token 成本控制

| 循环级别 | 每次循环额外 Token | 最大循环次数 | 最大额外成本 |
|---------|------------------|------------|------------|
| D1 | ~50 token | 3 | ~150 token |
| D2 | ~100 token | 2 | ~200 token |
| D3 | ~200 token | 1 (人工) | ~200 token |

## 四、Hook+Loop 与 System Prompt 的联动

### 在 System Prompt 中的定义

```
TraeOrchestrator 主调度
  └── Hook 工程
       ├── Hook 准备：任务开始前检查
       │    └── 对应安全规则 + 资源检查 + Provider 检查
       ├── Hook 执行中：每阶段完成检查
       │    └── 对应进度/质量/风险/资源检查
       ├── Hook 验收：子任务完成检查
       │    └── 对应 D1/D2/D3 验收
       └── Hook 完成：全任务完成检查
            └── 对应完整检查 + 文档检查 + Token 审计
```

### 在 Skill 中的激活

```
Skill 执行过程中
  │
  ├── 自动注册 Hook 点
  │    ├── ccs_provider_switch → Hook 1：安全检查
  │    │    └── 检查目标 Provider 是否可用
  │    ├── ccs_mcp_sync → Hook 1：权限检查
  │    │    └── 检查是否有同步权限
  │    └── token_stats_budget → Hook 3：预算检查
  │         └── 检查预算值是否合理
  │
  └── 完成时触发验收
       └── D1 阶段验收 → 结果反馈
```

## 五、Hook+Loop 与三大技能的集成

### cc-switch 中的 Hook

```
ccs_provider_switch
  ├── Hook 1：检查目标 Provider 是否可用
  │    └── 调用 ccs_provider_test 验证连通性
  ├── Hook 2：检查当前任务是否可中断
  │    └── 如有正在执行的任务，等待完成
  ├── Hook 3：验证切换后 API 连通性
  │    └── 调用 ccs_provider_test 验证新 Provider
  └── Hook 4：记录切换日志
       └── token-stats 记录切换事件

  Loop：切换失败
  └── 回退到上一个 Provider
  └── 记录失败模式
  └── 最多重试 2 次
```

### token-stats 中的 Hook

```
token_stats_budget 设置
  ├── Hook 1：检查预算值是否合理
  │    └── 预算值不能为负数，不能超过月上限
  ├── Hook 2：检查是否覆盖所有 Provider
  │    └── 所有 Provider 的预算分配总和正确
  ├── Hook 3：验证告警阈值配置
  │    └── 阈值在 0-100% 之间
  └── Loop：如果超额
       └── 触发告警 → 自动切换便宜 Provider
       └── 记录超额事件 → 生成告警报告
```

### master-engineer-hub 中的 Hook

```
路由决策
  ├── Hook 1：检查引力盆是否命中
  │    └── Level 0/1 匹配检查
  ├── Hook 2：检查 Agent 负载
  │    └── 目标 Agent 是否可接受新任务
  ├── Hook 3：路由结果验证
  │    └── 路由到的 Agent 是否有能力执行
  └── Loop：路由失败
       └── 回退 L3 兜底 → 重路由
       └── 最多重试 2 次
       └── 记录路由失败模式
```

## 补充说明

### 1. Hook 节点与三大技能的映射

| Hook 节点 | master-engineer-hub | cc-switch | token-stats |
|----------|-------------------|-----------|-------------|
| Hook 1 | 路由前准备检查 | Provider 可用性 | 预算初始检查 |
| Hook 2 | 路由执行状态 | Provider 运行状态 | Token 实时监控 |
| Hook 3 | 路由结果验证 | 切换结果验证 | 预算合规审计 |
| Hook 4 | 路径固化到引力盆 | 切换日志记录 | **最终 Token 审计** |

### 2. Loop 循环的引力盆学习

每次 Loop 循环记录的失败原因会被送入引力盆，用于未来避免同样错误：

```
引力盆失败模式记录：
{
  "pattern": "ccs_provider_switch",
  "failure_reason": "目标 Provider API 不可用",
  "suggestion": "切换前先调用 ccs_provider_test",
  "hit": 2
}
```

当同类失败到达阈值（3 次）时，系统会自动调整路由策略：
- 增加 `ccs_provider_test` 作为切换的前置步骤
- 将不可用的 Provider 标记为"不推荐"
- 自动切换到备用 Provider

### 3. Hook 超时处理

| Hook 节点 | 超时时长 | 超时处理 |
|----------|---------|---------|
| Hook 1 | 5s | 默认允许执行（宽松策略） |
| Hook 2 | 10s | 记录警告，继续执行 |
| Hook 3 | 15s | D1 默认通过，D2/D3 需等待 |
| Hook 4 | 5s | 默认完成（不阻塞交付） |

### 4. 全程监工 5 项指标

Hook+Loop 体系全程监控 5 项核心指标：

| 指标 | 监控方式 | 告警阈值 | 超限处理 |
|------|---------|---------|---------|
| 进度 | PlanAgent 跟踪 | 偏差 > 20% | 调整计划 |
| 质量 | CodeReviewAgent 审查 | 缺陷 > 3 个 | Loop 修正 |
| 成本 | token-stats 审计 | 预算 > 80% | 切换便宜 Provider |
| 安全 | SecurityAgent 监控 | 任何高危操作 | 立即拦截 |
| 时间 | TaskAgent 计时 | 超时 > 30s | 超时处理 |
