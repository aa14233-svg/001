# Hook+Loop 工程约束与验收体系

> King Agent 不是"放手不管"，而是"全程监督"——通过 Hook 和循环约束确保每一步都符合标准。

## 一、Hook 工程体系

### 4 个核心 Hook 节点

```
任务开始 → Hook 1（准备）→ 执行 → Hook 2（执行中）
              │                              │
              ▼                              ▼
          安全检查                    进度/质量/风险
              │                              │
              ▼                              ▼
          资源确认                    阶段性检查
              │                              │
              ▼                              ▼
          依赖检查                    纠正/调整
              │                              │
              ▼                              ▼
       ┌─────┴─────┐               ┌─────────┴─────────┐
       │           │               │                   │
       ▼           ▼               ▼                   ▼
   Hook 3（验收）   完成        Hook 4（完成）         交付
       │                           │
       ▼                           ▼
   D1/D2/D3 验收              完整检查
       │                           │
       ▼                           ▼
   通过/驳回                  文档/结果
```

### Hook 1：任务准备

| 检查项 | 内容 | 执行者 |
|--------|------|--------|
| 安全检查 | 路径是否在禁区，是否有高危操作 | SecurityAgent |
| 资源确认 | 文件是否存在，依赖是否满足 | SearchAgent |
| 权限验证 | 是否有权限执行该操作 | SecurityAgent |
| 上下文确认 | 是否理解任务上下文 | ChatAgent |

### Hook 2：执行中

| 检查项 | 内容 | 执行者 |
|--------|------|--------|
| 进度检查 | 是否按计划推进 | PlanAgent |
| 质量检查 | 产出是否符合标准 | CodeReviewAgent |
| 风险检查 | 是否有偏离风险 | TaskAgent |
| 资源检查 | 预算是否充足 | AuditAgent |

### Hook 3：验收

| 检查项 | 内容 | 执行者 |
|--------|------|--------|
| D1 验收 | 阶段子任务完成度 | PlanAgent |
| D2 验收 | 模块功能完整性 | CodeReviewAgent |
| D3 验收 | 全任务总体验收 | AuditAgent |

### Hook 4：完成

| 检查项 | 内容 | 执行者 |
|--------|------|--------|
| 完整检查 | 所有子任务是否完成 | TaskAgent |
| 文档检查 | 是否生成了必要的文档 | DocAgent |
| 清理检查 | 临时文件是否已清理 | DeployAgent |

## 二、三级验收体系（D1/D2/D3）

### D1：阶段验收

```
D1 验收标准
  │
  ├── 当前阶段任务全部完成
  ├── 产出物经 Agent 自查
  ├── 无重大错误或偏离
  └── 通过 → 进入下一阶段
      拒绝 → 返工修正
```

**调用时机**：每个子任务完成后

### D2：模块验收

```
D2 验收标准
  │
  ├── 相关模块的所有子任务完成
  ├── 模块间接口正常
  ├── 模块功能符合预期
  ├── 模块内无严重缺陷
  └── 通过 → 进入下一模块
      拒绝 → 缺陷修复
```

**调用时机**：相关功能模块完成后

### D3：总体验收

```
D3 验收标准
  │
  ├── 全任务所有子任务完成
  ├── 所有模块集成正常
  ├── 最终产出符合原始需求
  ├── 所有 Hook 检查均通过
  ├── Token 消耗在预算内
  └── 通过 → 交付
      拒绝 → 回归修复
```

**调用时机**：全任务完成后

## 三、Loop 循环机制

```
当验收不通过时，触发 Loop 循环：

D1 拒绝 → 修正 → 重新 D1 验收
  │
  ├── 最多 3 次循环
  ├── 每次循环记录失败原因
  └── 3 次后升级到 D2 审查

D2 拒绝 → 修复 → 重新 D2 验收
  │
  ├── 最多 2 次循环
  └── 2 次后升级到 D3 审查

D3 拒绝 → 重大问题 → 人工介入
```

## 四、Hook+Loop 与 System Prompt 的联动

### 在 System Prompt 中的定义

```
TraeOrchestrator 主调度
  └── Hook 工程
       ├── Hook 准备：任务开始前检查
       │    └── 对应安全规则 + 资源检查
       ├── Hook 执行中：每阶段完成检查
       │    └── 对应进度/质量/风险检查
       ├── Hook 验收：子任务完成检查
       │    └── 对应 D1/D2/D3 验收
       └── Hook 完成：全任务完成检查
            └── 对应完整检查 + 文档检查
```

### 在 Skill 中的激活

```
Skill 执行过程中
  │
  ├── 自动注册 Hook 点
  │    ├── ccs_provider_switch → Hook 安全
  │    ├── ccs_mcp_sync → Hook 权限
  │    └── token_stats_budget → Hook 预算
  │
  └── 完成时触发验收
       └── D1 阶段验收 → 结果反馈
```

## 五、Hook+Loop 与三大技能的集成

### cc-switch 中的 Hook

```
ccs_provider_switch
  ├── Hook 1：检查目标 Provider 是否可用
  ├── Hook 2：检查当前任务是否可中断
  ├── Hook 3：验证切换后 API 连通性
  └── Hook 4：记录切换日志
```

### token-stats 中的 Hook

```
token_stats_budget 设置
  ├── Hook 1：检查预算值是否合理
  ├── Hook 2：检查是否覆盖所有 Provider
  ├── Hook 3：验证告警阈值配置
  └── Loop：如果超额 → 触发告警 → 自动切换
```

### master-engineer-hub 中的 Hook

```
路由决策
  ├── Hook 1：检查引力盆是否命中
  ├── Hook 2：检查 Agent 负载
  ├── Hook 3：路由结果验证
  └── Loop：路由失败 → 回退 L3 兜底 → 重路由
```
