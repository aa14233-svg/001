## 三、拓扑层次总结

| 层次 | 名称 | 核心组件 | 职责 | Token 成本 |
|------|------|---------|------|-----------|
| L1 | 入口层 | TraeOrchestrator + System Prompt | 接收请求，角色定义 | ~2KB |
| L2 | Hook 层 | 4 个 Hook 节点 | 安全约束，质量检查 | ~50 token |
| L3 | 路由层 | master-engineer-hub + 引力盆 | 0 token 路由 A/B/C | 0-200 token |
| L4 | 执行层 | 20+ Agent 集群 | 具体任务执行 | 取决于任务 |
| L5 | 基础设施 | LSO-DAO 五境递进 | 修炼体系，递进执行 | ~100 token |
| L6 | 技能层 | SkillHub + 8 大技能组 | 专业能力提供 | ~50 token |
| L7 | 验收层 | D1/D2/D3 + Loop | 质量验收，迭代改进 | ~100 token |

## 四、数据流路径

```
用户输入（自然语言）
  │
  ├── ● 路由路径（高频 90%）
  │     引力盆/特征词典命中 → 直通对应 Agent → 执行 → 返回
  │     Token 成本：0
  │     示例："切换到大模型" → 引力盆命中 → ccs_provider_switch
  │
  ├── ● 标准路径（中频 8%）
  │     LLM 兜底 → 路由 A/B/C → 五境递进 → SkillHub → 验收 → 返回
  │     Token 成本：中等（~300 token）
  │     示例："写一个 React 页面" → 完整编码流程
  │
  └── ● 扩展路径（低频 2%）
       LLM 兜底 → 能力缺口 → 机械降神 → GitHub 检索 → 注入 → 执行 → 消散
       Token 成本：高（~1000+ token）
       示例："帮我用新工具 X 做 Y" → 按需注入工具
```

### 数据流路径的 Token 效率对比

| 路径类型 | 占比 | 平均 Token | 最大 Token | 延迟 |
|---------|------|-----------|-----------|------|
| 路由路径 | 90% | **0** | 0 | <5ms |
| 标准路径 | 8% | ~300 | ~2000 | ~500ms |
| 扩展路径 | 2% | ~1000 | ~5000 | ~2s |

## 五、拓扑中的关键路由决策点

```
整个拓扑中有 3 个关键路由决策点：

决策点1：[入口层] 基础 System Prompt 加载
  ├── 决策：基础还是增强？
  ├── 条件：任务复杂度
  └── 影响：Agent 能力范围

决策点2：[路由层] master-engineer-hub 路由
  ├── 决策：Level 0/1/2/3 哪一级？
  ├── 条件：引力盆命中率 + 特征匹配度
  └── 影响：Token 成本（0 vs 全量）

决策点3：[执行层] Agent 归属路由
  ├── 决策：A/B/C 哪个 Agent？
  ├── 条件：任务类型 + Agent 可用性
  └── 影响：工具可见性 + 执行效率
```

## 六、拓扑的容错机制

| 故障点 | 容错策略 | 恢复时间 | 影响范围 |
|--------|---------|---------|---------|
| TraeOrchestrator 宕机 | 自动重启 | <1s | 当前请求失败 |
| Provider 不可用 | cc-switch 自动切换 | <200ms | 仅当次调用 |
| Agent 加载失败 | 使用替代 Agent | <100ms | 当前子任务 |
| 引力盆数据损坏 | 重建空引力盆 | <50ms | Level 0/1 降级到 L3 |
| Hook 超时 | 默认通过 | 0ms | 跳过当前检查 |
| 机械降神失败 | 返回"能力不足" | <500ms | 当前请求 |

## 补充说明

### 1. 拓扑中的三大技能位置

| 技能 | 在拓扑中的位置 | 核心作用 |
|------|--------------|---------|
| master-engineer-hub | L3 路由层 | 唯一的入口网关，决定"谁来执行" |
| cc-switch | L4A 路由管理层 | Provider 切换和 MCP 管理 |
| token-stats | L4A + L7 | 贯穿执行全过程的 Token 审计和预算管理 |
| lso-os | L5 基础设施层 | 五境递进的修炼体系 |
| omnipotent-engineer | L4B 工程执行层 | 全栈开发和工程交付 |

### 2. 拓扑的数据流方向

```
主数据流（从上到下）：
用户输入 → L1 → L2 → L3 → L4 → L5 → L6 → L7 → 交付

反馈数据流（从下到上）：
交付结果 → L7 → L6 → L5 → L4 → L3 → L1 → 引力盆更新

控制数据流（贯穿所有层）：
token-stats 审计 ← 贯穿 L1-L7
Hook 检查 ← 贯穿 L2-L7
引力盆固化 ← 从 L3 出发，影响 L1-L7
```

### 3. 拓扑的扩展点

| 扩展点 | 位置 | 说明 |
|--------|------|------|
| 新增 Skill | L6 技能层 | 添加新的 SKILL.md 文件，注册到 SkillHub |
| 新增 Agent | L4 执行层 | 在 Agent 集群中添加新 Agent，定义归属域 |
| 新增 Provider | L4A | 通过 cc-switch 注册新的 Provider |
| 新增 Hook | L2/L7 | 在现有 Hook 链中插入新的检查节点 |
| 新增路由规则 | L3 | 在 master-engineer-hub 中添加新的特征模式 |

### 4. 拓扑的监控指标

| 层次 | 监控指标 | 告警条件 |
|------|---------|---------|
| L1 | 请求量、响应时间 | QPS > 100 或 P99 > 1s |
| L2 | Hook 通过率 | 通过率 < 90% |
| L3 | Level 0/1 命中率 | 命中率 < 80% |
| L4 | Agent 负载、错误率 | 错误率 > 5% |
| L5 | 五境执行时间 | 单境 > 10s |
| L6 | 技能调用成功率 | 成功率 < 95% |
| L7 | D3 通过率、Token 预算 | 预算 > 80% |
